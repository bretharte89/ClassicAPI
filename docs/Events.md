# Vanilla 1.12.1 Event Table

Snapshot of the engine's event-name table (`[VAR_EVENT_TABLE_BASE_PTR]`,
`0x00CEEF68`) as captured via `_classicapi_DumpAllEvents()` on a 1.12.1 client with only ClassicAPI. Reflects what `frame:RegisterEvent` will accept at runtime in that configuration.

Indices are the engine's runtime event IDs (the values
`Event::Custom::Lookup` returns and the dispatcher's
`mov eax, eventID * 0x10` arithmetic uses). Gaps in the numbering are
NULL slots — the table's input array at `0x00BE1198` has ~200 unwritten
entries scattered through the range that the engine's
`RebuildEventTable` preserves as NULLs. NULL slots are what ClassicAPI
and other DLLs claim for custom events.

## Notes on specific entries

- **Repeated `UNIT_*` entries at low indices** (e.g. four slots named
  `UNIT_DAMAGE` at 128–131, five `UNIT_STATS` at 144–148, seven
  `UNIT_RESISTANCES` at 149–155) — these are the per-stat /
  per-resistance / per-damage-school broadcast events. They share a
  name because the engine routes them to the same dispatch path;
  addons registering `UNIT_DAMAGE` get hits from any of the four
  slots.
- **`[543] LOTTERY_SHOW` and `[548] LOTTERY_ITEM_UPDATE`** — both
  stock Blizzard events, strings at VAs `0x008514E8` and `0x00851474`
  in the unmodified 1.12.1 binary, written into the input array by
  `FUN_0051AB30` (the event-table initializer). Neither has a known
  public feature in vanilla, which makes them natural targets for
  other DLLs to repurpose: SuperWoWhook overwrites slot 548's
  `.rdata` string in-place to `UNIT_CASTEVENT`, and Turtle WoW's
  binary replaces slot 543's content with `CHAT_MSG_HARDCORE`. See
  [CLAUDE.md § Other DLLs can inject events by patching `.rdata`
  strings](../CLAUDE.md#other-dlls-can-inject-events-by-patching-rdata-strings-in-place).
  On a stock-binary client neither override fires — those are the
  events you see only on the respective patched build.
- **`[534] START_MINIGAME` / `[535] MINIGAME_UPDATE`** — both stock
  Blizzard events, verified by finding the strings at VAs
  `0x00851584` / `0x00851574` in an unmodified 1.12.1 binary. The
  binary even contains a build-path debug string
  `"E:\build\buildWoW\WoW\Source\Minigame\Minigame_C.cpp"`, so
  Blizzard had an actual `Minigame_C` source module compiled in —
  the events are real engine surface, just for a system that never
  shipped publicly.
- **`[545] QUEST_WATCH_UPDATE`, `[544] CHAT_MSG_FILTERED`,
  `[546] CHAT_MSG_BATTLEGROUND`, `[547] CHAT_MSG_BATTLEGROUND_LEADER`**
  — all stock Blizzard, verified present in the 1.12.1 binary's
  strings (`0x008514c0`, `0x008514d4`, `0x008514a8`, `0x00851488`).
  Several of these were added between 1.11 and 1.12 — e.g.
  `CHAT_MSG_FILTERED` is not in the 1.11 strings — so the
  1.11→1.12 patch did expand the event set, which means absence
  from a 1.11 binary isn't sufficient to prove an event is Turtle.
- **Gaps in numbering** (1, 3–15, 31–39, 42–106, 108–119, etc.) —
  NULL slots in the table. These are claimable by DLLs running on
  this client. ClassicAPI claims 11 of them for its custom events;
  `VanillaMinimapTracking` claims one for `MINIMAP_UPDATE_TRACKING`;
  SuperWoWhook (when present) likewise claims slots for its
  injections.

## Regenerating

Run `/script _classicapi_DumpAllEvents()` in-game with ClassicAPI
loaded. Output goes to `WTF\Account\<acct>\debug.log`. To regenerate
this doc after a binary update, dump fresh and strip the 11 entries
listed in [API.md § Events](API.md#events) (plus
`MINIMAP_UPDATE_TRACKING` if VMT was loaded during the capture).
